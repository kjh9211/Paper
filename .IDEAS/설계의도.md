기획서 잘 읽었습니다. Java로 외부 JAR(Spark)를 실행·관리하고, Redis Pub/Sub으로 Heartbeat를 모니터링하는 Aether 오케스트레이터 설계를 작성해 드릴게요.핵심 클래스들을 파일로 작성할게요.총 4개 파일로 설계했습니다. 전체 구조를 정리하면:

---

## 📐 클래스 구조 및 설계 의도

### `SparkProcess.java` — JAR 실행 & PID 관리
- `ProcessBuilder`로 외부 JAR를 자식 프로세스로 실행하며, `SPARK_ID`, `SPARK_PORT`, `REDIS_HOST` 등 환경 변수를 주입합니다.
- Java 9+의 `process.pid()`와 `ProcessHandle.of(pid)`를 이중으로 사용해 PID 추적의 신뢰도를 높입니다.
- 종료 시 SIGTERM → 10초 대기 → SIGKILL 순서로 우아한 종료를 보장합니다.

### `HeartbeatMonitor.java` — Redis Pub/Sub 감시 핵심
Pub/Sub과 폴링을 **이중 레이어**로 구성한 것이 핵심입니다.

| 레이어 | 방식 | 역할 |
|---|---|---|
| 1차 | `channel:heartbeat` 구독 (Pub/Sub) | 실시간 alive 수신 |
| 2차 | `spark:registry:{id}` Hash 폴링 | Pub/Sub 네트워크 문제와 실제 사망 구분 |

이렇게 하면 Redis 연결이 일시 끊겼다고 Spark를 무조건 재시작하는 오탐을 방지합니다. Dead 임계값은 `DEAD_THRESHOLD_SEC = 15`초 (Heartbeat 주기 5초의 3배)로 설정했습니다.

### `AetherOrchestrator.java` — 메인 오케스트레이터
- `handleDeadSpark()`: 10초 **재시작 쿨다운**을 두어 연속 재시작 루프를 차단합니다.
- `handleConflict()`: `WATCH` → `MULTI/EXEC` 트랜잭션으로 Conflict 중재 시 원자성을 보장합니다. `resolveConflict()`는 `protected`로 오버라이드 가능하게 두었습니다.
- Builder 패턴으로 `sparkCount`, `basePort`, `jarPath` 등을 유연하게 설정합니다.

### `SparkHeartbeatPublisher.java` — Spark 측 구현체
- `HSET spark:registry:{id} heartbeat_ts <epoch>` + `PUBLISH channel:heartbeat {id}:alive`를 5초마다 발행합니다.
- `channel:orchestration` 구독으로 Aether의 `cmd:shutdown`, `cmd:sync` 명령을 수신합니다.

## ⚠️ 다음 단계에서 고려할 점

**Conflict 중재 로직** (`resolveConflict`)은 현재 Last-Write-Wins 스텁으로 두었습니다. 게임 로직에 따라 연산값 합산, 타임스탬프 비교, 또는 외부 규칙 엔진 호출 방식으로 교체가 필요합니다.

**의존성** (build.gradle 또는 pom.xml에 추가 필요):
```
redis.clients:jedis:5.x
```
