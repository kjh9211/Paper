# **\[기획서\] Project Aether: Redis 기반 분산 서버 오케스트레이션 시스템**

## **1\. 프로젝트 개요**

본 프로젝트는 중앙 관리 구동기인 \*\*'Aether(에테르)'\*\*가 하위의 여러 **'Spark(스파크)'** 인스턴스들을 동적으로 생성하고 제어하며, Redis를 통해 연산 데이터의 일관성을 유지하는 고성능 분산 서버 아키텍처 구축을 목표로 함.

## **2\. 시스템 네이밍 및 역할**

### **2.1 Aether (Main Orchestrator)**

* **정의:** 시스템의 '뇌' 역할을 하는 최상위 구동기.  
* **역할:** \- Spark 인스턴스들의 생명 주기(기동, 감시, 종료) 관리.  
  * Redis를 통한 데이터 충돌 발생 시 최종적인 연산 중재 및 데이터 확정.  
  * 전역 설정(Global Config) 관리 및 하위 배포.

### **2.2 Spark (Child Instances)**

* **정의:** 실제 게임 로직이 돌아가는 개별 Paper 서버 인스턴스.  
* **역할:**  
  * 독립적인 비즈니스 로직 및 사용자 요청 처리.  
  * Redis에 연산 결과 반영 시도(Optimistic Lock 사용).  
  * 충돌 발생 시 Aether에게 알림(Pub/Sub) 후 명령 대기.

### **2.3 Nexus (Redis Shared Layer)**

* **정의:** Aether와 Spark들이 소통하고 데이터를 공유하는 매개체.

## **3\. 핵심 기능 명세**

### **3.1 계층적 프로세스 관리**

* **Spawn & Control:** Aether는 설정된 수만큼의 Spark를 독립된 포트와 환경 변수를 부여하여 실행함.  
* **Heartbeat Monitoring:** Spark는 주기적으로 자신의 상태를 Redis에 갱신하고, Aether는 이를 감시하여 장애 발생 시 즉시 해당 Spark를 재구동함.

### **3.2 중재 시스템 (Conflict Resolution)**

* **1단계 (Spark):** 연산 수행 후 Redis의 version 값을 확인하며 저장을 시도함.  
* **2단계 (Collision):** 만약 다른 Spark가 먼저 데이터를 수정했다면 저장에 실패하고, Aether에게 event:conflict 신호를 보냄.  
* **3단계 (Aether):** Aether는 충돌된 데이터를 직접 읽어 최종 로직에 따라 연산값을 확정하고 Redis를 업데이트함. 이후 Spark들에게 갱신된 데이터를 동기화하라는 신호를 보냄.

### **3.3 우아한 시스템 종료 (Graceful Exit)**

* 관리자가 Aether를 종료하면, Aether는 모든 Spark에게 종료 명령을 전달함.  
* Spark들이 모든 데이터를 Redis에 안전하게 쓰고 연결을 해제할 때까지 기다린 후 전체 프로세스를 종료함.

## **4\. 데이터 설계 (Redis Schema)**

* aether:config: 전체 시스템 공통 설정값 (JSON)  
* spark:registry:{id}: 각 Spark의 PID, 포트, 업타임 정보 (Hash)  
* nexus:data:state: 공유되는 게임/연산 데이터 본체 (String/JSON)  
* nexus:data:version: 낙관적 락을 위한 데이터 버전 정보 (Integer)  
* channel:orchestration: Aether와 Spark 간의 명령 및 충돌 보고 채널 (Pub/Sub)

## **5\. 기대 효과**

* **무한 확장성:** 서버 부하에 따라 Aether 명령 한 번으로 Spark 인스턴스를 자유롭게 증설 가능.  
* **데이터 정합성:** 분산 환경의 고질적 문제인 데이터 덮어쓰기 문제를 Aether의 중앙 중재로 완벽히 해결.  
* **복구 자동화:** 하위 서버(Spark)가 죽더라도 상위 관리기(Aether)가 즉시 복구하여 서비스 중단 최소화.
